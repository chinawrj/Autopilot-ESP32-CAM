## Completed Work

### WebSocket Control Messages

- **set_quality**: Client sends `{"action":"set_quality","value":N}` (4-63), immediately applies to camera sensor
- **set_resolution**: Client sends `{"action":"set_resolution","value":"QVGA|VGA|SVGA|XGA"}`
- **get_status**: Triggers immediate heartbeat push (fps, clients, heap_free, heap_min)
- **Architecture decision**: Control messages only modify settings + set flags, no response sent from handler (avoids socket contention with stream push task)
  - All WS send operations centralized in streaming task (Core 1)
  - Uses `volatile bool s_hb_now` flag to trigger immediate heartbeat

### Heartbeat Mechanism

- Auto-pushes JSON text frame to all connected clients every 5 seconds
- Content: `{"type":"heartbeat","fps":N,"clients":N,"heap_free":N,"heap_min":N}`
- get_status command triggers immediate heartbeat (no 5-second wait)

### Heap Monitoring

- `/api/status` new fields: `heap_free`, `heap_min`
- WS heartbeat includes the same heap data
- Current heap: ~4125KB free, ~4107KB minimum

### Frontend Update (stream_udp.html)

- HUD now shows HEAP display (updated from heartbeat)
- Quality dropdown: Q:10 Best / Q:12 / Q:20 / Q:30 / Q:50 Fast
- Resolution dropdown: QVGA / VGA / SVGA / XGA
- Status bar shows clients + heap details

### Bug Fix

- **WS socket send race condition**: Handler attempted `httpd_ws_send_frame()` while streaming task called `httpd_ws_send_frame_async()` → errno 119 + connection dropped
- Solution: Control message handler no longer sends responses; instead sets `s_hb_now` flag for streaming task to send uniformly
